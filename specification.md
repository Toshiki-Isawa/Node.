## 1. 概要

### 1.1 プロダクト概要

Node. は「植物の時間を残す」をコンセプトとした、植物観測アーカイブアプリである。植物の観測・成長・育成履歴を蓄積し、定点観測・成長比較・タイムライン・タイムラプス・育成履歴・観測習慣を通じて、「植物を観測する体験そのもの」を価値化する。

### 1.2 プロダクトの本質

Node. は「植物を管理するアプリ」ではなく、「植物の時間を記録するアプリ」である。最終的には **植物観測 OS** として進化し、**植物の時間データ基盤** となることを目指す。

### 1.3 最重要思想

Node. は「植物を育てるアプリ」ではなく、**「植物の時間を残すアプリ」** である。

---

## 2. プロダクト思想

### 2.1 Observation First

Node. の中心は **Observation（観測）** である。すべての機能設計・UX 判断は、観測体験の質を最優先とする。

### 2.2 提供価値

| 価値 | 説明 |
|------|------|
| 成長記録 | 植物の成長を継続的に記録できる |
| 時間変化の比較 | 任意の時点間で変化を視覚的に比較できる |
| 育成履歴の蓄積 | 観測データが時系列で積み上がり、資産となる |
| 観測習慣の形成 | 定点観測が自然な習慣として定着する |
| 時間の資産化 | 植物固有の「時間」が長期的な価値を持つ |

### 2.3 競合との差別化

| 一般的な植物アプリ | Node. |
|-------------------|-------|
| 水やり通知中心 | 観測中心 |
| AI 診断単発 | 長期履歴蓄積 |
| 管理ツール | 観測アーカイブ |
| SNS 投稿 | 個人記録 |
| Todo 感 | ギャラリー感 |

Node. は **AI 診断アプリ・SNS・水やり通知アプリ** ではない。差別化の核は **観測履歴** である。

## 8. 技術構成

### 8.1 アーキテクチャ

| レイヤー | 技術 |
|----------|------|
| Client | SwiftUI |
| Local DB | SwiftData |
| Backend | Supabase |
| Storage | Cloudflare R2 |
| Analytics | PostHog |
| Ads | Google AdMob |
| Video（タイムラプス） | AVFoundation（端末内生成。クラウド非保存） |

### 8.2 画像・Archive 戦略

Node. の価値は **「植物の時間を積み上げること」** にある。課金は機能ではなく **保存品質・Archive・Original・長期履歴** を中心に設計する（12.2 参照）。

| 保存先 | 役割 | 方針 |
|--------|------|------|
| ローカル（Original） | 撮影直後の即時保存 | 同期完了まで保持。同期後は R2 正本へ退避し端末から削除 |
| ローカル（サムネイル） | オフライン閲覧 | 常駐。Timeline / Collection 表示に使用 |
| クラウド（R2） | 長期 Archive 正本 | Seed: 圧縮版 / Archive・Conservatory: Original |
| ローカル（キャッシュ） | 一時取得 | Compare / Timelapse 利用時に R2 からオンデマンド取得 |
| タイムラプス MP4 | 端末内の一時生成物 | **保存しない**（Export 先はユーザー任意。インフラコスト回避） |

| プラン | ローカル Original | ローカルサムネイル | クラウド保存 | クラウド容量 |
|--------|-------------------|-------------------|--------------|--------------|
| Seed | 同期完了まで保持 | 常駐 | 圧縮版のみ | 3 GB |
| Archive | 同期完了まで保持 | 常駐 | Original 保持 | 50 GB（調整可能） |
| Conservatory | 同期完了まで保持 | 常駐 | Original 保持 | 500 GB（調整可能） |

詳細は 11.5（クラウド同期・Archive）・12.2（プラン構成）を参照。

### 8.3 Google AdMob

Node. の広告配信基盤として **Google AdMob** を採用する。配置方針は 12.4 に従い、**Rewarded Ads のみ** を使用する。

#### 8.3.1 SDK / 連携

| 項目 | 方針 |
|------|------|
| SDK | Google Mobile Ads SDK for iOS |
| 導入方式 | Swift Package Manager |
| 対象 OS | iOS 16 以上（MVP 方針に準拠） |
| 使用フォーマット | Rewarded のみ |
| 非使用フォーマット | Banner、Interstitial、App Open、Native |

#### 8.3.2 広告ユニット管理

| 環境 | 方針 |
|------|------|
| 開発 / TestFlight | Google 提供のテスト広告ユニット ID を使用 |
| 本番 | AdMob コンソールで発行した Rewarded 広告ユニット ID を使用 |
| 設定管理 | Debug / Release で ID を切り替え。本番 ID はソースコードに直書きしない |

#### 8.3.3 プライバシー / コンプライアンス

| 項目 | 方針 |
|------|------|
| ATT | iOS App Tracking Transparency に対応。必要に応じて許可ダイアログを表示 |
| UMP | Google User Messaging Platform（GDPR / EEA 向け同意管理）に対応 |
| SKAdNetwork | Info.plist に AdMob 推奨の SKAdNetwork ID を設定 |
| プライバシーポリシー | 広告配信・第三者 SDK 利用を明記 |
| App Store Privacy Nutrition Labels | AdMob に準拠したデータ収集項目を申告 |

#### 8.3.4 実装方針

- `AdMobService`（仮称）として広告ロジックを Domain 層から分離する
- Premium ユーザーは SDK 呼び出し自体を行わない
- Export フロー開始前に Rewarded 広告をプリロードする
- 広告表示・視聴完了・失敗・スキップ不可等のイベントを PostHog に送信する

#### 8.3.5 エラーハンドリング

| 状況 | 方針 |
|------|------|
| 広告読み込み失敗 | リトライ UI を表示 |
| リトライ上限到達 | Export を解放し、UX 阻害を最小化 |
| 視聴未完了 | Export は解放しない |
| オフライン | 「ネットワーク接続が必要」旨を表示し、Export は保留 |

### 10.3 Camera UX

**最重要機能。** Node. の価値の中心。

#### 要件

| 要件 | 詳細 |
|------|------|
| 画面向き | 縦固定 |
| 起動速度 | カメラを素早く起動 |
| 撮影テンポ | 連続撮影のテンポを損なわない |

#### 撮影フロー

```
1. 植物選択
2. カメラ起動
3. 撮影
4. 即保存
5. （任意）ログ追加
```

- ローディングを見せない
- 通信待ちさせない
- 撮影テンポを止めない
- ログ入力を強制しない
- 保存を即時完了に感じさせる

- 植物が主役
- UI は最小限
- 撮影行為に集中

#### 撮影補助機能

- グリッド
- 前回写真オーバーレイ
- 定点補助

### 10.7 Timelapse（タイムラプス）

#### MVP 実装

Phase 1（Closed Beta）で実装する。

**理由：**

- 価値が一瞬で伝わる
- SNS 共有と相性が良い
- 継続モチベーションになる

#### 基本方針（インフラ・保存）

| 項目 | 方針 |
|------|------|
| 生成場所 | **端末内のみ**（`AVFoundation` 等） |
| 入力素材 | ローカルに保存済みの観測画像（Original / サムネイル） |
| クラウド保存 | **行わない**（R2・Supabase へ動画をアップロードしない） |
| 恒久保存 | アプリ内 DB には保持しない。Export 時に写真ライブラリ保存または共有シートでユーザーが任意保存 |
| サーバー生成 | **採用しない**（FFmpeg ワーカー・Edge Function による生成は廃止方針） |

インフラコスト増大を避け、観測画像（既存の Local-first 資産）だけで成果物を作る。

#### 出力形式

| 項目 | 方針 |
|------|------|
| コンテナ | MP4（H.264 等、端末がサポートするコーデック） |
| Seed Export | 最大 720p |
| Archive / Conservatory Export | 最大 4K（端末性能・ソース解像度の範囲内） |
| アスペクト | Instagram / TikTok 向けプリセット（将来。MVP は横長またはソースに準拠） |

#### 技術方針（iOS）

| 項目 | 方針 |
|------|------|
| フレームソース | 選択した Observation のローカル画像を時系列で並べる |
| 最小枚数 | 5 枚以上（`TimelapseRequirements.minimumObservations`） |
| 最大フレーム数 | 60 枚（超過分は間引きまたは末尾優先で採用） |
| 処理 | バックグラウンドでエンコード。UI は進捗表示 |
| 完了後 | プレビュー → Export（共有シート / 写真ライブラリ） |
| ネットワーク | 生成・Export に **必須ではない**（オフライン観測データのみで完結可能） |

#### 非要件（明示的にやらないこと）

- タイムラプス動画のクラウド同期・アーカイブ
- サーバーサイドのタイムラプス生成 API・ジョブテーブル
- 生成動画の Supabase / R2 上の長期保管
- 機種変更時のタイムラプス動画のクラウド復元

#### マネタイズ連携

- 無料ユーザー（Seed）：端末内生成完了後、Google AdMob の Rewarded Ad を表示し、Export を解放（8.3・12.4 参照）
- 有料ユーザー（Archive / Conservatory）：広告なし・即 Export・高解像度（4K）出力

#### 受け入れ基準

- 5 枚以上の観測がある植物で、ネットワークなしでも端末内生成が開始できる
- 生成された MP4 を共有シートまたは写真ライブラリへ Export できる
- タイムラプス動画が R2 / `storage_objects` に登録されない
- Seed は Export 前に Rewarded Ad、Archive 以上は広告なしで Export できる

### 10.13 AdMob 連携

Phase 2（Public MVP）で Google AdMob を導入する。詳細な配置方針は 12.4、技術要件は 8.3 を参照。

#### 要件

| 要件 | 詳細 |
|------|------|
| 広告 SDK | Google AdMob（Google Mobile Ads SDK for iOS） |
| 広告形式 | Rewarded Ads のみ |
| 配置対象 | Timelapse Export（MVP 時点の唯一の広告ポイント） |
| Premium 除外 | Premium ユーザーには広告を一切表示しない |
| プリロード | Export フロー前に Rewarded 広告を事前読み込み |
| 視聴完了 | 視聴完了後に Export を解放 |
| 計測 | 表示・視聴完了・失敗・リトライを PostHog で計測 |

#### 表示フロー

```
1. 無料ユーザーが Timelapse Export を開始
2. タイムラプス生成中 UI を表示
3. Rewarded 広告をプリロード
4. 生成完了後、Rewarded 広告を表示
5. 視聴完了 → Export 解放
6. 視聴未完了 → Export 保留
```

#### 受け入れ基準

- 無料ユーザーが Timelapse Export 時に Rewarded 広告を視聴完了すると Export できる
- Premium ユーザーは広告なしで即 Export できる
- Camera、Timeline、Observation 保存フロー中に広告が表示されない
- Banner / Interstitial がアプリ内に存在しない
- 広告読み込み失敗時にリトライ UI が表示される
- ATT / UMP が必要な地域・条件で適切に同意フローが表示される
- Debug ビルドではテスト広告ユニットが使用される

#### 非要件（MVP 時点）

- Banner 広告
- Interstitial 広告
- Comparison Share Export 時の Rewarded 広告（将来拡張候補）

## 11. 非機能要件

### 11.1 パフォーマンス

- カメラ起動は体感待ち時間を最小化すること
- 撮影後の保存処理はユーザー操作をブロックしないこと
- 保存を即時完了に感じさせること

### 11.2 オフライン / 同期（Local First）

#### 方針

Observation は **ローカル保存優先** とする。

| 方式 | 詳細 |
|------|------|
| Background Sync | バックグラウンドで同期 |
| Queue 方式 | 同期キューによる順次処理 |
| 非同期アップロード | ユーザー操作をブロックしない |

#### UX 思想

**「通信を意識させない」**

Local First は **端末への永久フル保存** を意味しない。撮影後は即ローカル Original を保存し、**同期完了後は R2 を正本として Original を端末から退避** する。サムネイルは端末に常駐し、Compare / Timelapse 等で Original が必要な場合は R2 からオンデマンド取得する。クラウド同期・容量管理の詳細は 11.5 を参照。

### 11.3 可用性

- オフライン環境でも観測・撮影・Quick Log が利用可能
- 同期失敗時もローカルデータは保持され、再試行可能

### 11.4 セキュリティ

- ユーザーデータは認証済みユーザーのみがアクセス可能
- 画像データは適切なアクセス制御のもと Cloudflare R2 に保存

### 11.5 クラウド同期・Archive

#### 基本思想

Node. は長期利用によって価値が増すプロダクトである。Seed ユーザーにも最低限のクラウド同期を提供し、**「安心して積み上げられる」** 状態を作る。

| 原則 | 説明 |
|------|------|
| 履歴は消さない | 一定期間での履歴削除・古い Observation の消去は行わない |
| 差別化軸 | 履歴期間ではなく **保存品質** と **保存容量** で Seed / Archive / Conservatory を差別化 |
| 観測体験優先 | 容量超過でも撮影・ローカル保存は継続。同期のみ停止 |
| クラウド正本 | 同期完了後、長期 Archive の正本は R2 とする |
| 端末 Original 退避 | 同期成功後、Original ファイルを端末から削除。サムネイルは保持 |
| オンデマンド取得 | Compare / Timelapse 等で Original が必要な場合、R2 から presigned URL で一時取得 |

#### 同期戦略

| 方式 | 詳細 |
|------|------|
| Local First | 撮影後は即ローカル保存。同期はバックグラウンド実行 |
| Seed 同期 | Seed ユーザーにもクラウド同期を提供（圧縮版・3 GB 上限） |
| Queue 方式 | 同期キューによる順次処理 |
| 非同期アップロード | ユーザー操作をブロックしない |

#### Seed — クラウド保存

| 項目 | 方針 |
|------|------|
| 保存内容 | 圧縮画像、サムネイル、Observation メタデータ |
| 容量上限 | 3 GB |
| 履歴保持期間 | 無制限 |

#### Archive — クラウド保存

| 項目 | 方針 |
|------|------|
| 保存品質 | Original（HEIF / 高解像度） |
| 容量上限 | 50 GB（調整可能） |
| タイムラプス動画 | クラウド保存しない（端末内生成・ユーザー Export のみ） |
| 復元 | 観測画像・メタデータの完全復元（タイムラプス MP4 は対象外） |

#### Conservatory — クラウド保存

| 項目 | 方針 |
|------|------|
| 保存品質 | Original |
| 容量上限 | 500 GB（調整可能） |
| 将来価値 | RAW 保持、Lineage、Ownership History |

#### 容量上限到達時

| 動作 | 詳細 |
|------|------|
| 新規 Observation | ローカル保存継続 |
| クラウド同期 | 停止（`sync_paused_storage_limit`） |
| アプリ利用 | 継続可能 |
| 上位プラン導線 | Archive / Conservatory への静かな案内 |

#### Seed クラウド圧縮品質

| 項目 | 方針 |
|------|------|
| 最大辺 | 2048 px |
| JPEG 品質 | 0.72 前後 |
| 目的 | Compare / Timelapse に十分な画質を確保しつつ容量を抑制 |

#### UX 原則（禁止事項）

- 撮影中・保存前の広告
- 同期失敗による保存ブロック
- 容量超過による撮影停止

#### 許容する挙動

- 同期のみ停止
- ローカル保存継続
- 後から Archive / Conservatory へ移行可能

#### 将来拡張（Post-MVP）

**Smart Compression（Free 向け）**

| 期間 | 品質 |
|------|------|
| 直近 30 日 | 高品質 |
| 30 日以降 | 高圧縮 |

**将来の Premium 価値:** RAW 保存、Plant Passport、Lineage 履歴、Transfer 完全同期、AI 履歴分析

## 12. マネタイズ戦略

### 12.1 基本思想

Node. のマネタイズは **観測体験を阻害しない** ことを最優先とする。Node. は **機能課金** ではなく、**時間の蓄積** — 保存品質・Archive・Original・長期履歴 — を中心に課金価値を設計する。

| プラン | コンセプト | 方針 |
|--------|-----------|------|
| Seed | 観測を始める | 観測体験を無料提供。クラウド同期（圧縮・3 GB）。Export 時 Rewarded Ads |
| Archive | 植物の時間を残す | Original 同期・50 GB・広告なし Export |
| Conservatory | コレクションを保存する | Archive 全機能 + 500 GB + 将来のコレクター機能 |

広告は **「観測体験の対価」** ではなく、**「成果物出力コスト」** として扱う。

### 12.2 プラン構成（松竹梅）

### 12.4 広告戦略（Monetization Ads Strategy）

#### 12.4.1 基本方針

Node. の価値は **Observation（観測）** そのものにある。以下の導線では広告を表示しない。

- 撮影中
- 保存中
- タイムライン閲覧中

#### 12.4.2 禁止する広告配置

**観測導線内広告** は一切禁止する。

| 禁止配置 | 理由 |
|----------|------|
| 撮影直後全画面広告 | 撮影テンポを阻害する |
| Timeline スクロール途中広告 | 観測体験を中断する |
| Observation 保存前広告 | 保存フローを遅延させる |
| Camera 起動前広告 | 観測開始を妨げる |
| 強制 Interstitial | 「静かな観測体験」を毀損する |

Node. のブランド価値は **「静かな観測体験」** である。広告によるテンポ阻害は UX 毀損リスクが非常に高い。

#### 12.4.3 採用する広告モデル：Rewarded Ads

Node. の広告は **「成果物出力」** にのみ配置する。

| 項目 | 方針 |
|------|------|
| 広告 SDK | Google AdMob |
| 広告形式 | Rewarded Ads（リワード広告） |
| 最重要配置 | Timelapse Export |
| 原則 | 「撮る行為」に広告を入れない |

**Timelapse Export が最重要広告ポイントである理由：**

1. **待機時間が自然発生する** — 端末内エンコード・画像合成により「生成待ち」が自然に存在する
2. **ユーザー心理と一致する** — Observation 時は「記録したい」モード、Timelapse Export 時は「成果物を得たい」モード。「広告を見て解放」と相性が良い
3. **Premium 導線になる** — 生成・Export 頻度の増加に伴い、「広告なし価値」を感じやすくなる

#### 12.4.4 Timelapse Export フロー

**無料ユーザー：**

```
1. タイムラプス生成開始
2. 「生成中」表示
3. Rewarded Ad 表示
4. Export 解放
```

**Premium ユーザー：**

- 広告なし
- 即 Export
- 高画質生成（4K）

#### 12.4.5 UX 原則

| 原則 | 説明 |
|------|------|
| 撮る行為に広告を入れない | 観測フローは常に無広告 |
| 成果物生成のみ | 広告表示対象は Export 系フローに限定 |
| ブランド整合性 | 「広告アプリ」に見えてはいけない |

#### 12.4.6 将来的な広告拡張候補

| 候補 | 内容 |
|------|------|
| Comparison Share Export | Before / After 画像生成、SNS 共有画像生成 |
| 高画質 Export | 高解像度出力時 |

#### 12.4.7 将来的スポンサー候補

Node. のユーザー層（植物コレクター、LED 育成ユーザー、園芸用品購入層）との相性を活かし、以下とのスポンサー連携余地がある。

- LED
- 鉢
- 用土
- 肥料
- 園芸イベント

#### 12.4.8 ブランド整合性

広告は **「静かな観測体験」** を維持した上で、**「成果物生成フロー」** に自然に統合する。Node. は **「広告アプリ」** に見えてはいけない。

### 12.5 将来的価値

育成履歴の資産化 — 長期間蓄積された観測データ自体が価値を持つ。
